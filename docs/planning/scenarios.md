# Auth Board - 사용자 시나리오

## 시나리오 1: 회원가입

```
1. 사용자가 회원가입 페이지에 접속한다
2. 이메일, 비밀번호, 닉네임을 입력한다
3. [검증] 이메일 형식 확인, 중복 이메일 확인
4. [검증] 비밀번호 최소 8자, 영문+숫자 포함
5. 비밀번호를 BCrypt로 암호화하여 DB에 저장한다
6. role은 USER로 기본 설정된다
7. 회원가입 완료 → 로그인 페이지로 이동
```

## 시나리오 2: 로그인

```
1. 사용자가 이메일, 비밀번호를 입력한다
2. [검증] 이메일로 회원 조회 → 없으면 "이메일 또는 비밀번호가 올바르지 않습니다"
3. [검증] BCrypt로 비밀번호 비교 → 불일치 시 동일 에러 메시지
4. Access Token (30분) 발급 → 응답 body에 포함
5. Refresh Token (7일) 발급 → HttpOnly Cookie로 설정 + DB 저장
   - 기존 Refresh Token이 있으면 삭제 후 새로 저장
6. 메인 페이지로 이동
```

## 시나리오 3: 토큰 갱신

```
1. 클라이언트가 API 요청 시 Access Token이 만료됨 (401 응답)
2. Axios 인터셉터가 자동으로 /api/auth/refresh 호출
   - Cookie에 담긴 Refresh Token이 함께 전송됨
3. [검증] Refresh Token이 DB에 존재하는지 확인
4. [검증] Refresh Token 만료 여부 확인
5. 새 Access Token 발급 → 응답 body에 포함
6. 원래 요청을 새 Access Token으로 재시도
```

## 시나리오 4: 로그아웃

```
1. 사용자가 로그아웃 버튼을 클릭한다
2. 서버에 /api/auth/logout 요청
3. DB에서 해당 사용자의 Refresh Token 삭제
4. HttpOnly Cookie 제거 (Set-Cookie 만료 처리)
5. 클라이언트에서 Access Token 메모리 초기화
6. 로그인 페이지로 이동
```

## 시나리오 5: 비로그인 사용자 - 글 조회

```
1. 비로그인 사용자가 메인 페이지에 접속한다
2. 글 목록이 표시된다 (제목, 작성자 닉네임, 작성일)
3. 글을 클릭하면 상세 페이지로 이동한다
4. 글 내용이 표시된다
5. "글 작성" 버튼은 비활성화 또는 클릭 시 로그인 페이지로 이동
```

## 시나리오 6: 로그인 사용자 - 글 작성

```
1. 로그인한 사용자가 "글 작성" 버튼을 클릭한다
2. 제목, 내용을 입력한다
3. [검증] 제목 필수, 최대 100자 / 내용 필수
4. 작성 요청 시 Access Token이 헤더에 포함된다
   - Authorization: Bearer {accessToken}
5. 서버가 토큰에서 회원 정보를 추출하여 게시글 저장
6. 작성 완료 → 글 상세 페이지로 이동
```

## 시나리오 7: 본인 글 수정/삭제

```
[수정]
1. 본인이 작성한 글의 상세 페이지에서 "수정" 버튼이 보인다
2. 수정 페이지에서 제목/내용을 변경한다
3. 서버에서 토큰의 회원 ID와 게시글의 member_id 비교
4. 일치 → 수정 성공
5. 불일치 → 403 Forbidden

[삭제]
1. 본인이 작성한 글의 상세 페이지에서 "삭제" 버튼이 보인다
2. 삭제 확인 다이얼로그 표시
3. 확인 → 삭제 요청
4. 서버에서 토큰의 회원 ID와 게시글의 member_id 비교
5. 일치 → 삭제 성공 → 목록 페이지로 이동
6. 불일치 → 403 Forbidden
```

## 시나리오 8: 관리자 - 글 삭제

```
1. ADMIN 역할의 사용자가 아무 글의 상세 페이지에 접속한다
2. 본인 글이 아니어도 "삭제" 버튼이 보인다
3. 삭제 요청 시 서버에서 role이 ADMIN인지 확인
4. ADMIN → 삭제 성공
```

## 시나리오 9: 관리자 - 회원 관리

```
1. ADMIN이 관리자 페이지에 접속한다
   - USER 역할로 접근 시 403 Forbidden
2. 전체 회원 목록이 표시된다 (이메일, 닉네임, 가입일, 역할)
3. 특정 회원의 "강제 탈퇴" 버튼 클릭
4. 확인 다이얼로그 표시
5. 확인 → 해당 회원 삭제 + 해당 회원의 게시글도 삭제
6. ADMIN 자기 자신은 강제 탈퇴 불가
```

## 시나리오 10: 인가 실패 케이스 정리

| 상황 | 응답 |
|------|------|
| Access Token 없이 인증 필요 API 호출 | 401 Unauthorized |
| 만료된 Access Token으로 API 호출 | 401 Unauthorized → 토큰 갱신 시도 |
| Refresh Token도 만료됨 | 401 Unauthorized → 로그인 페이지 이동 |
| USER가 관리자 API 호출 | 403 Forbidden |
| 타인의 글 수정/삭제 시도 | 403 Forbidden |
| 탈퇴된 회원의 토큰으로 요청 | 401 Unauthorized |
